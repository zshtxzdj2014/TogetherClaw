# TogetherClaw

多平台会话统一管理工具 - 像聊天室一样管理多个 Claw 平台。

## 摘要

TogetherClaw 是一个多 Claw 平台 API 并发调用管理器，让你能够通过 API Keys 同时连接多个 Claw 平台，在统一的 Web 界面中管理所有对话。

TogetherClaw 就像一个聊天应用，让你可以同时与多个平台对话，在标签页之间自由切换，对比不同平台的回答，就像把四个人放在同一个房间，可以同时和他们对话。

## 问题陈述

使用多个 Claw 平台时面临的实际问题：

1. **需要打开多个网页或软件**：每个平台都要单独登录和管理，窗口切来切去很麻烦。

2. **无法同时对话**：想对比不同平台的回答，或者让不同平台处理不同任务，需要手动切换。

3. **对话会丢失**：切换平台时，之前的对话历史可能找不到了。

4. **无法对比效果**：想知道哪个平台回答得更好，需要分别提问再手动对比。

TogetherClaw 通过统一的 Web 界面和多会话管理解决这些问题。

### 核心组件

- **连接管理器**：管理所有平台的 API Keys 和连接配置
- **会话管理器**：管理每个平台的对话历史和上下文
- **API 客户端**：统一的 API 调用接口，适配不同平台
- **消息路由器**：处理消息的发送、广播和响应聚合

## 使用场景

### 1. 多平台并行对话

同时与多个平台对话，每个平台独立会话：

```
在 ArkClaw 标签页：
User: "帮我审查这段代码的安全性"
ArkClaw 正在回答...

切换到 Linclaw 标签页：
User: "为这个项目生成 API 文档"
Linclaw 正在回答...

两个对话同时进行，互不干扰
```

### 2. 平台效果对比

把同一个问题发给多个平台，对比回答：

```
点击"广播"按钮，输入：
"优化这段代码的性能"

在界面上看到：
┌─────────────────────────────────┐
│ ArkClaw 的回答：                │
│ 建议使用缓存...                 │
├─────────────────────────────────┤
│ Linclaw 的回答：                │
│ 建议使用异步处理...             │
├─────────────────────────────────┤
│ DuClaw 的回答：                 │
│ 建议优化数据库查询...           │
└─────────────────────────────────┘

选择最好的回答使用
```

### 3. 切换不中断

在标签页之间自由切换，对话历史保持：

```
在 ArkClaw 标签页聊了 10 条消息
切换到 Linclaw 标签页聊了 5 条消息
再切回 ArkClaw 标签页
→ 之前的 10 条消息还在，可以继续对话
```

### 4. 多账号管理

同一平台可以添加多个账号：

```
添加：
- ArkClaw (工作账号)
- ArkClaw (个人账号)
- Linclaw (测试账号)
- Linclaw (生产账号)

在不同场景使用不同账号
```

## 配置

### 方式 1: 通过 Web 界面配置（推荐）

1. 启动应用后，点击"添加平台"
2. 选择平台类型（ArkClaw、Linclaw 等）
3. 输入 API Key
4. 点击"测试连接"
5. 保存配置

### 方式 2: 通过配置文件

创建配置文件 `~/.switchclaw/config.yaml`：

```yaml
platforms:
  - id: arkclaw-main
    name: "ArkClaw (字节跳动)"
    type: arkclaw
    api_endpoint: "https://ark.cn-beijing.volces.com/api/v3"
    api_key: "ark_xxxxxxxxxxxxxxxx"
    model: "default-model"

  - id: linclaw-main
    name: "Linclaw (七牛云)"
    type: linclaw
    api_endpoint: "https://api.qiniu.com/linclaw/v1"
```

## 工作原理

TogetherClaw 通过 API Keys 调用云服务，不需要本地部署：

1. **连接管理**：保存每个平台的 API Key（加密存储）
2. **API 调用**：通过 HTTPS 调用各平台的 API 端点
3. **会话隔离**：每个平台独立的对话会话和历史记录
4. **消息路由**：支持单发、多发、广播等多种发送方式
5. **本地存储**：对话历史保存在本地，随时查看

## 技术栈

- **前端**: React + TypeScript + Tailwind CSS
- **后端**: Node.js + Express (可选，也可以纯前端)
- **存储**: 本地 JSON 文件
- **加密**: API Keys 使用 AES-256 加密存储
- **通信**: HTTPS API 调用

## 系统要求

- **Node.js**: v18 或更高版本
- **操作系统**: Windows、macOS、Linux
- **浏览器**: Chrome、Firefox、Safari、Edge (最新版本)
- **网络**: 需要访问已配置平台的 API 端点

## 安全性

- **API Key 加密存储**: 使用 AES-256 加密算法
- **HTTPS 通信**: 所有 API 调用使用 HTTPS
- **本地运行**: 应用运行在本地，数据不上传到任何服务器
- **会话隔离**: 每个平台的会话独立，不会混淆
- **权限控制**: 配置文件权限设置为仅所有者可读写

## 常见问题

### Q: 这些 Claw 平台需要本地部署吗？

A: 不需要。本项目通过 API Keys 与平台接口通信，无需本地部署平台服务。

### Q: 可以同时与多个平台对话吗？

A: 可以。每个平台独立的对话会话，在标签页之间切换即可。

### Q: 切换平台时对话会中断吗？

A: 不会。切换只是切换查看的标签页，每个平台的对话都保持着。

### Q: 最多可以添加多少个平台？

A: 理论上无限制。你可以添加任意数量的平台，甚至同一平台的多个账号。

### Q: 如何对比不同平台的回答？

A: 使用"广播"功能，把同一个问题发给所有平台，然后在界面上对比回答。

### Q: API Keys 安全吗？

A: API Keys 使用 AES-256 加密存储在本地，应用本身也运行在本地，不会上传到任何服务器。

### Q: 支持流式响应吗？

A: 计划在 Phase 4 支持，让回答像打字一样逐字显示。

### Q: 可以导出对话历史吗？

A: 计划在 Phase 3 支持，可以导出为 Markdown 或 JSON 格式。
